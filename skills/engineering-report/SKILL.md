## 一句话

这个 skill 覆盖工程的完整生命周期：**先追问校验实现计划，再固化工程文档**。不写设计推导长文。

## 适用场景

- 代码写完后，整理工程落地文档
- 实现前，制定可执行的实现计划
- 迭代中，根据代码变更同步工程文档
- 部署前，过一遍实现完整性

## 与设计报告的边界

| 维度 | 工程报告 | 设计报告 |
|---|---|---|
| 核心问题 | 怎么实现、在哪里、如何验证 | 为什么这样设计 |
| 主要证据 | 真实代码、配置、测试、日志 | 问题、约束、方案、原则 |
| 关注对象 | 文件、接口、流程、集成、观测 | 推导、算法、取舍 |
| 更新时间 | 频繁 | 相对低频 |

设计报告是工程报告的上游输入。工程报告要对齐设计，但不能重复抄设计。遇到设计相关内容，**链接设计报告**，不重新写。

## 第一部分：逐条追问

在写工程文档之前，必须先通过追问校验实现的完整性和与设计的一致性。

### 追问原则

1. **一次只问一个问题**，等用户回答后再问下一个
2. **能读代码就读代码**——只问用户代码里找不到答案的问题
3. **校验一致性**——实现是否与设计报告对齐？不对齐时立刻暴露
4. **暴露风险**——实现中有哪些隐式约定、技术债、未覆盖的边界？

### 追问顺序

```text
1. 实现范围（改了什么？没改什么？）
       ↓
2. 与设计的对齐（设计里的决策是否真的落地？有没有偷工减料？）
       ↓
3. 接口与契约（对外暴露什么？调用方需要满足什么前提？）
       ↓
4. 测试覆盖（测了什么？没测什么？关键风险是否覆盖？）
       ↓
5. 可观测性（出问题后靠什么定位？日志/指标/报告是否齐全？）
```

### 与设计报告的交叉校验

追问过程中必须检查四件事：

1. 设计中的关键决策是否真的落地到代码
2. 设计里的约束是否在工程上被体现
3. 设计里延后的项，工程是否偷偷实现了一半
4. 工程里新增的隐式约定，是否需要反写回设计报告

### 追问结束条件

- 实现范围明确（改了哪些文件、哪些模块）
- 与设计报告的对齐情况已确认（一致 / 有差异需记录）
- 测试和可观测性的覆盖情况已清楚

### 追问产物

追问结束后，输出一份简短的**工程准备结论**：

1. 实现范围一句话定位
2. 与设计报告的对齐情况
3. 需要记录的工程层面决策
4. 测试和观测缺口
5. 文档归属建议（应新建/更新的路径清单）

## 第二部分：文档分层与工程落点

工程正文默认写在 **`engineering/`**（如 `implementation_vN.md`、`known_issues.md`），不要把工程报告写成第二份设计文档。

| 内容 | 写入 | 禁止 |
|---|---|---|
| 代码路径、命令、diff、测试结果 | `engineering/` | 长篇方案对比 |
| 与设计不一致 | `engineering/` 中专节 + 提醒反刷 `design/`、`adr/` | 在工程报告重新定义架构 |
| 字段语义 / JSON 契约 | 摘要 + **链接** `schema/` | 复制整份 schema |
| 全局取舍未落地 | 提醒新增/更新 `adr/` | 仅在 Decision 口头描述 |

### 目录职责

| 目录 | 写什么 | 不写什么 |
|---|---|---|
| `architecture/` | 本 skill 不展开（交给 design-report） | |
| `schema/` | 仅引用或短摘要，不复制 | |
| `design/` | 本 skill 不展开（交给 design-report） | |
| `adr/` | 提醒更新，但不在此写 ADR 正文 | |
| `engineering/` | 工程落点、命令、结果、与设计差异 | 候选方案矩阵、设计推导 |

## 第三部分：工程文档标准结构

### 九章结构

```text
1. 实现目标
2. 模块落点
3. 数据结构
4. 执行流程
5. 配置契约
6. API / 契约
7. 下游集成
8. 测试与可观测性
9. 开放问题
```

### 章节跳过规则

| 章节 | 可跳过条件 | 跳过写法 |
|---|---|---|
| 1. 实现目标 | 设计报告已有，且实现完全对齐 | "实现目标见 [设计报告链接]，本文不重复。" |
| 3. 数据结构 | 无新增或修改的数据结构 | "本次实现不涉及新数据结构。" |
| 5. 配置契约 | 无可调参数变更 | "本次实现不涉及新配置。" |
| 6. API / 契约 | 无新增或修改的对外接口 | "本次实现不涉及 API 变更。" |
| 7. 下游集成 | 无下游消费方变更 | "本次实现不影响下游。" |
| 9. 开放问题 | 无遗留问题 | "本次实现无开放问题。" |

**判断标准：如果某章写出来只有 2-3 行且不增加信息量，就跳过。**

### 目标长度

工程文档主体目标 **≤ 120 行**（不含 YAML 头、图表、代码片段）。

超过时检查：是否混入了设计推导（→ design/）、是否可跳过某些章节。

### 快速定位卡

工程报告最上面必须先给"快速定位卡"，让读者不用翻全文：

1. 系统位置：一行 pipeline 图
2. 输入 → 输出：模块级数据流图
3. 关键决策：简短列表（链接设计报告）
4. 下游影响：谁消费、改了会影响谁

图优先，文字只保留必要说明。

## 第四部分：各章生成规则

### 第 1 章：实现目标

要回答：
- 这个模块解决什么问题（链接设计报告）
- 本次实现目标是什么，不是什么
- MVP 边界

**不重复写设计报告已有的问题定义和约束分析。** 用一句话 + 链接即可。

### 第 2 章：模块落点

要回答：
- 代码在哪些文件
- 每个文件职责是什么
- 哪些文件是入口，哪些是共享层

要求：用树形图 + 文件职责表，每个文件一句话描述。

### 第 3 章：数据结构

要回答：
- 核心结构是什么
- 哪些字段来自 LLM、配置、确定性计算或外部系统
- 哪些字段最容易被误解

与 `schema/` 的关系：如果字段契约已在 `schema/` 中定义，这里只写实现差异和注意事项，链接 schema。

### 第 4 章：执行流程

这是工程报告最重要的章节。

要回答：
- 从入口到出口经历了什么步骤
- 哪些地方短路、重试、降级、兜底
- 哪些状态变化是隐式完成的

推荐拆法：入口 → 核心步骤 → 错误处理/兜底 → 输出与收尾。

**必须画 plaintext 流程图。**

### 第 5 章：配置契约

要回答：
- 有哪些可调参数
- 默认值和风险是什么
- 环境变量、配置文件、代码常量如何映射

如果某参数调错会"悄悄变差但不报错"，要显式标红说明。

### 第 6 章：API / 契约

要回答：
- 对外暴露什么接口
- 调用方必须满足什么前提
- 可能抛什么错，幂等性如何

### 第 7 章：下游集成

要回答：
- 谁消费这个模块
- 消费哪个字段、哪个函数、哪个文件产物
- 改输出格式会波及谁

要特别关注"静默出错"：不报错，但下游含义已经变了。

### 第 8 章：测试与可观测性

合并两个关注点：

**测试部分**：
- 当前有哪些测试
- 哪些关键风险还没测
- 测试分层是否合理

最少覆盖：正常路径、边界输入、失败兜底、契约回归。

**可观测性部分**：
- 线上或离线跑批时，如何知道它出问题了
- 出问题后靠什么定位
- 报告、日志、指标、trace 分别承载什么职责

### 第 9 章：开放问题

只写工程层面的遗留：
- 已知技术债
- 待验证假设
- 暂时接受的 workaround
- 依赖外部确认的问题

**不重复设计报告第 8 章的延后项。** 只补充工程实现中发现的新问题。

## 第五部分：图示规则

工程报告的图要优先帮助定位代码和影响范围。

### 必须画图的场景

| 场景 | 图类型 |
|---|---|
| 快速定位卡 | 系统位置图 + 输入输出图 |
| 执行流程 | plaintext 流程图（入口→核心→错误→输出） |
| 多模块交互 | 模块调用图 |

### plaintext 图风格

- 用 `┌──┐`、`│`、`▼`、`──▶` 等 ASCII 字符
- 每个框一句话职责
- 箭头标注调用方向或数据流

---

## 第六部分：增量更新规则

更新已有工程报告时，不要整篇重写。先定位变动，再更新对应章节。

常见映射：

| 代码变化 | 优先更新章节 |
|---|---|
| 数据模型/结构体变化 | 3, 6, 7 |
| 执行逻辑变化 | 4, 8 |
| 配置变化 | 5, 8 |
| 文件结构变化 | 2 |
| 接口变化 | 6, 7, 8 |
| 测试变化 | 8 |
| 日志/监控变化 | 8 |

更新后必须附一行变动摘要。

## 第七部分：与设计报告的交接规则

如果已有设计报告，工程报告要显式检查这四件事：

1. 设计中的关键决策是否真的落地到代码
2. 设计里的约束是否在工程上被体现
3. 设计里延后的项，工程是否偷偷实现了一半
4. 工程里新增的隐式约定，是否需要反写回设计报告

差异记录格式：

```markdown
> **与设计差异: 标题**
>
> - **设计预期**：设计报告中的描述
> - **工程实际**：代码中的实际实现
> - **原因**：为什么有差异
> - **影响**：是否需要更新设计报告
```

